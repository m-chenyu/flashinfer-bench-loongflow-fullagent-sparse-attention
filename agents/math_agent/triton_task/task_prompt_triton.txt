3. **Implementation Restrictions (Strictly Enforced)**:
    * **No Black Boxes**: Implement all mathematical operations (matmul, softmax, normalization, etc.) yourself using Triton primitives (`tl.load`, `tl.store`, `tl.dot`, `tl.sum`, `tl.max`, `tl.exp`, etc.).

    * **Pointer Arithmetic**: Use stride-based pointer arithmetic for multi-dimensional tensor access.

# Mandatory Innovation Dimensions (Architectural-Level Changes)
Your solution must demonstrate a **substantial performance improvement** over a naive implementation. You must innovate in at least **one** of the following dimensions:

1. **Algorithm & Parallelization Strategy**:
    * **Operation Fusion**: Fuse multiple operations (e.g., scale + softmax + matmul) into a single kernel pass to eliminate intermediate HBM reads/writes.
    * **Tiling**: Tile the computation along multiple dimensions to maximize data reuse within a block.
    * **Reduction Optimization**: Use `tl.sum`, `tl.max` with correct axis/mask handling, or implement hierarchical reductions instead of naive serial loops.

2. **Memory Access Patterns**:
    * **Coalesced Global Memory**: Ensure that threads within a warp access contiguous memory addresses to maximize memory bandwidth.
    * **Vectorized Loads/Stores**: Load 128-bit vectors where possible by choosing block sizes divisible by 4 (for float32) or 8 (for float16).
    * **Shared Memory (via registers)**: In Triton, thread-local tiles are stored in registers; design `BLOCK_M x BLOCK_K` tiles carefully to maximize register reuse.

3. **Hardware-Specific Optimization**:
    * **Tensor Cores**: Use `tl.dot(a, b)` with appropriately shaped 2D tiles (multiples of 16) to leverage Tensor Core instructions for massive compute throughput.
    * **Software Pipelining**: Use `tl.multiple_of` hints and structure loops for prefetching overlap.
    * **Mixed Precision**: Cast to `tl.float16` or `tl.bfloat16` for Tensor Core paths while accumulating in `tl.float32`.

4. **Control Flow & Launch Configuration**:
    * **Block Size Tuning**: Choose `BLOCK_SIZE` as a power of 2 (64, 128, 256, 512) to maximize occupancy.
    * **Grid Mapping**: Map program IDs to output tiles in a way that maximizes L2 cache reuse (e.g., swizzled tile ordering).
    * **Masked Loads**: Always use `mask=` parameters in `tl.load`/`tl.store` for boundary-safe access.

```python
import torch
import torch.nn as nn
import triton
import triton.language as tl


@triton.jit
def elementwise_scale_add_kernel(
    a_ptr, b_ptr, out_ptr,
    alpha,
    n_elements,
    BLOCK_SIZE: tl.constexpr,
):
    """Fused kernel: out = alpha * a + b, vectorized over n_elements."""
    pid = tl.program_id(0)
    offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n_elements

    a   = tl.load(a_ptr   + offsets, mask=mask)
    b   = tl.load(b_ptr   + offsets, mask=mask)
    out = alpha * a + b

    tl.store(out_ptr + offsets, out, mask=mask)


class ModelNew(nn.Module):
    def __init__(self, alpha: float) -> None:
        super().__init__()
        self.alpha = alpha

    def forward(self, a, b):
        out = torch.empty_like(a)
        n_elements = a.numel()
        BLOCK_SIZE = 1024
        grid = (triton.cdiv(n_elements, BLOCK_SIZE),)
        elementwise_scale_add_kernel[grid](
            a, b, out,
            self.alpha,
            n_elements,
            BLOCK_SIZE=BLOCK_SIZE,
        )
        return out
```
